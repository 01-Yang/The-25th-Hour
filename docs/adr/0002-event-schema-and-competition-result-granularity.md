# 事件字段颗粒度与竞赛结果落库口径

在“第二十五小时”当前版本里，我们补充确认两组会直接影响实现配置的数据口径。

第一，事件文档虽然继续按“普通随机事件 / 交互事件 / 模型周事件 / 万里路事件”维护文案，但开发配置不再只保留最小结果字段，而是统一收口为同一套结构化字段：`event_id / title / pool / phase / sentiment / tags / context_tags / base_weight / cooldown_weeks / repeatable / ai_related / ai_experience_delta / trigger_rules / result / options / note`。这样可以保证事件文案、触发限定、权重冷却、AI 标记和数值结算都挂在同一条事件记录上，不再出现规则层写一套、程序层再简化出第二套命名的分叉。首版实现可以只先消费最小必填字段完成抽取与结算，但字段口径本身不再拆成两套。

第二，竞赛结果正式收口为“每学期最多投稿 1 次 + 单作品唯一投稿 + 图纸等级只做投稿门槛 + 按作品分判奖 + 奖金与简历分开结算 + 可随时查看获奖情况”。这里的“作品”默认指一份已完成的课程设计作业，因此同一作品只能投稿一次，也不能把同一课题重复投向多个竞赛。只要本学期尚未投稿且作品满足对应赛事门槛，就可以投稿。是否得奖只由该次投稿作品的 `作品分` 概率表判定；`S` 级作品直接获得一等奖，其他作品分区间都保留未获奖概率。获奖后给奖金，并把“赛事名 + 奖项”写入个人简历。竞赛结果不直接增加作品集，只服务于金钱收益、简历记录、结果查看和后续路线读取。玩家高频可见层只展示奖项结果文案与统一奖金提示，不再单独展示简历写入提示或竞赛成就提示。
