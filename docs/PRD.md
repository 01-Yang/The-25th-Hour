# 第二十五小时：建筑生模拟器 PRD

本文是项目的产品与架构总览，也承接开发前冻结口径。它只说明目标、边界、主系统关系和文档优先级；具体规则、数值、内容和技术实现分别由 `systems.md`、`numbers.md`、内容文档和 `technical-architecture.md` 维护。

## 产品定义

一款让玩家重新经历五年建筑学院青春的人生模拟器。

游戏不是纯建筑设计训练模拟器。建筑专业负责承载时间分配、压力、成长和人生分岔；玩家真正管理的是行动次数、精力、压力、金钱、课题成果和后期方向。

一句话气质：玩家不是在扮演“天才建筑师”，而是在五年的时间、金钱、精力、压力、图纸、路线选择和时代变化里，把一个普通学生慢慢推到毕业那天。

## 核心体验

- 每周只有固定行动机会，选择的重量来自取舍。
- 评图是每学期最核心的结算，进度回答“有没有做完”，作品质量回答“做得好不好”。
- 普通毕业应稳定可达，强路线保持稀有，中途失败有压迫但不高频。
- AI 是时代语境和后期路线条件，不是第二套主系统。
- 结局只读取 `1` 个主结局 + `1` 句状态收尾，不做多结局拼贴。
- 整体气质允许好笑，但底色不是纯搞笑；允许沉重，但不把玩家按进苦情。理想味道是：荒诞里有真实，狼狈里有一点温柔。
- Slogan：黄色的树林里不止分出两条路，少年，你的选择是什么呢？

## 文案原则

- 克制，不故意煽情。
- 真实，不把建筑学院写成纯爽文，也不把失败写成审判。
- 事件文本短，结局文本可以更完整，但都要留有余味。
- 文案可以嘴欠，但不能让系统逻辑变含糊。
- 涉及数值、路线和条件时，文本可以有文学性，规则必须清楚。
- 大一到大三少写大四焦虑和大五告别；大四、大五再逐渐加重路线感和回望感。

## 主结构

- `5` 年制，共 `60` 周。
- `1` 学年 = `2` 学期 = `12` 周。
- `1` 学期 = `6` 周，第 `5` 周为模型周，第 `6` 周为赶图周。
- 每周固定 `3` 次行动；商店道具 `星巴克会员周卡` 是唯一临时增加下周行动次数的例外。
- 每学期承载 `1` 个课题，只保留 `进度` 和 `作品质量` 两个课题临时值。
- 大五上和大五下共同承载同一个毕业设计，进度与作品质量连续继承。

## 游戏流程

1. 开局必须填写玩家昵称和大学名称；二者没有默认值，用于存档、称呼、剧情文本和排行榜展示，不提供数值加成。
2. 随机抽取 `3` 张角色卡，玩家可免费重抽 `1` 次并选择 `1` 个角色。
3. 进入开学典礼、军训、建院生活开始提示。
4. 军训结束后，从 `3` 位导师中选择 `1` 位。
5. 按周推进：触发阶段事件，执行行动，统一周结算，更新失败检查和高危状态。
6. 每学期末进入评图阶段，再做课程、GPA、作品集、竞赛等学期结算。
7. 大一下、大二下暑假触发固定顺序写生交互事件，不展开为暑假自由行动周。
8. 大五上第 `1` 周起全部正式参与入口常驻开放，不弹出任何提示或催促 UI；大五阶段完成报名、申请、投递、考试或确认后，才形成路线正式参与记录。
9. 大五下毕业设计答辩结束后，进入毕业典礼和结局结算；结尾动画方案当前不参与系统架构口径。

## 系统边界

- 角色：提供初始面板、被动特质和专属技能；不新增角色专属资源或额外行动槽。
- 导师：只影响阶段任务和事件语境；不设置认可度、好感度、独立关系条或隐藏路线分。
- 大学课程：轻量系统，只影响学期末答题、GPA 修正和事件语境。
- 事件：只保留普通随机事件、交互事件、模型周事件、万里路事件；课堂、导师、AI、地点只做语境标签。
- 建筑生的万里路：成长副系统，消耗行动和金钱，触发地点事件；不进入作品集。
- 暑假写生：固定节点，按指定顺序触发交互事件，只提供属性与事件收益，不扩成暑假自由行动周。
- 商店：轻量修正系统，不做库存、背包、耐久或 build 系统。
- 竞赛：读取已完成课题作品；奖金、简历记录和作品集分开结算。
- 实习：概率录取、持续周结算、写入简历并计入路线门槛；不占周行动槽。
- 后期路线：每局最多正式参与 `1` 个方向，正式参与后不跨路线补读结局。
- 结局：统一由结局解析器读取，其他系统不得提前写死人生结局。

## 成长结构

长期值：

- 角色六维
- 精力
- 压力
- 金钱
- 个人 GPA
- 作品集
- 个人简历
- AI 相关经历

学期临时值：

- 进度
- 作品质量

作品集只记录总分。最终评级为 `D` 及以上的课题按作品分入库；竞赛、实习、万里路、商店和课程不直接增加作品集总分。

## 后期路线

路线组固定为：

- 升学路线组
- 体制内路线组
- 建筑工作路线组
- 转行路线组

大一至大四不提供后期路线入口，也不记录任何后期路线状态。大五上第 `1` 周起，所有后期路线正式参与入口全部开放展示，但不弹出提示弹窗，不显示红点或催促 UI。只有大五阶段完成对应方向的报名、申请、投递、考试或确认后，才形成路线正式参与记录。

若玩家在大五上第 `1` 周不选择任何后期路线，系统不记录选择，也不做提醒；后续周次入口仍常驻可见，直到大五下毕业设计答辩开始后统一关闭。

若毕业前仍未正式参与方向，结局系统跳过路线结果，读取状态毕业结局。若已正式参与方向，结局只按已参与方向判定；失败后优先读取该方向专属回落，只有没有可读取方向专属回落时才读取状态毕业，不跳到其他路线补救。

## 失败与通关

只有完整读完五年、未触发中途失败且完成毕业设计，才算成功通关。

中途失败只保留少量明确条件：

- 生活费断裂
- 压力失控
- 连续挂科退学
- 被迫停学

结局读取优先级固定为：

1. 中途失败
2. 毕业失败——延毕
3. 已正式参与路线结果
4. 状态毕业

强路线只是高门槛成功结果，不是额外失败条件。

## 开发冻结

首版不再新增主系统。后续工作只进入四类：

- 实现落地：状态机、配置表、结算器和 UI 流程。
- 内容补全：事件、题库、结局文本、课程、角色和岗位内容。
- 数值调参：基于跑局模拟调整收益、门槛、权重和失败率。
- 体验打磨：调整展示顺序、提示文案、置灰逻辑和反馈节奏。

开发落地原则：

- 游戏状态必须有唯一真相源：统一 `GameState`。
- 时间推进必须使用显式状态机。
- 所有数值变化必须走统一结算器，并产出结构化 delta 与结算日志。
- 玩家每次行动后都应看到明确数值变化，不只给叙事反馈。
- 基础资源和长期属性透明展示；最终路线结果、考试答对数、内部折算分和后期路线内部判定结果不提前展示。
- 行动入口统一由可用性解析器返回 `可用 / 置灰 / 隐藏`。
- 每局生成固定随机种子，事件抽取、考试抽题、实习录取、留学申请和岗位投递内部判定都必须可复现。
- 事件、行动、路线、结局、成就等内容 ID 发布后保持稳定；展示文案不作为程序主键。
- 正式游戏界面、角色卡、存档展示、结局文本和排行榜展示中的角色名、学历背景、家庭背景必须显示中文；内部 ID 只服务存档、配置、统计和规则判断。
- 路线考试、申请、投递和内部判定结果进入内部状态缓存，不提前展示给玩家。
- 自动存档点固定为：每周开始、每周结束、评图前、评图后、路线正式参与后、结局结算前。
- 首版最小可玩闭环先保证固定 seed 下能从选角色跑到结局，再扩充 UI、事件和美术表现。

## 文档边界

- `systems.md` 是流程和系统关系源。
- `numbers.md` 是数值和结算源。
- `content-plan.md / events.md / endings.md / question-banks.md` 是内容池和展示文本源。
- 例外：导师阶段任务表的判定条件、奖励和处罚随导师文本维护在 `content-plan.md`，并由 `numbers.md` 明确引用。
- `technical-architecture.md` 记录技术实现边界与首版验收清单。

## 冲突处理

玩法、数值和内容冲突按以下顺序处理：

1. `PRD.md`
2. `systems.md`
3. `numbers.md`
4. 内容文档

技术实现冲突以 `technical-architecture.md` 为准，但不得覆盖 `PRD.md / systems.md / numbers.md` 已确定的玩法规则。
