# 技术架构决策

最后更新时间：`2026-05-27 +08:00`

## 目的

本文记录“第二十五小时：建筑生模拟器”的技术栈、存档、排行榜、部署和工程边界决策，用于后续开发前统一口径。

本文不定义玩法数值、事件内容或 UI 设计稿。若玩法规则与本文冲突，按 `PRD.md -> systems.md -> numbers.md -> 内容文档` 处理；若技术实现方式与本文冲突，以本文为准。

## 当前定案

- 首版以网站形式给玩家游玩，桌面网页体验优先。
- 同一套代码响应式适配移动端浏览器，但不以移动端为首发体验中心。
- 不做原生 App，不做 Unity / Godot / Phaser 游戏引擎路线。
- 不做登录，不做账号，不做云存档，不做身份找回。
- 本地存档存在浏览器中。
- 云端只保存总分排行榜所需的最小摘要数据。
- 排行榜只做总分榜，不做多榜单、好友榜、赛季榜或复杂反作弊。
- UI 未来要有舒服的动效体验，但当前阶段不做 UI 设计，也不写 UI 代码。

## 技术栈

推荐技术栈：

- 语言：`TypeScript`
- 前端构建：`Vite`
- 未来 UI 框架：`React`
- 未来动效：`GSAP`
- 本地存储：`IndexedDB`
- 轻量设置缓存：`localStorage`
- 部署：`Cloudflare Pages`
- 排行榜 API：`Cloudflare Workers`
- 排行榜数据库：`Cloudflare D1`

不采用：

- 不用 Unity / Godot / Phaser 承载首版。
- 不用复杂后端框架承载首版。
- 不在首版引入账号系统、云存档系统或后台管理系统。

## 模块边界

游戏核心必须独立于 UI。

推荐工程边界：

- `game-core`：状态机、规则、结算器、随机数、行动可用性、结局解析。
- `game-data`：行动、事件、课程、竞赛、实习、路线、结局等配置。
- `simulator`：自动跑局、验证、报告和调参工具。
- `web-app`：未来网页 UI，只展示状态并发起命令。
- `storage`：浏览器存档读写、版本迁移、本地玩家身份。
- `leaderboard`：总分提交、Top 100 读取、我的排名读取。

硬规则：

- UI 不直接修改 `GameState`。
- UI 只能调用游戏核心提供的命令接口。
- 所有数值变化必须走统一结算器，并产出结构化 delta 与结算日志。
- 随机结果必须来自统一 seed 和 RNG seam。
- 展示文案不作为程序主键；发布后的内容 ID 保持稳定。

## 本地存档

本地存档使用 `IndexedDB`。

存档内容：

- 当前 `GameState`
- 自动存档与手动存档元数据
- 已解锁人生结局
- 已解锁成长成就
- 历史重复达成人生结局记录
- 本地总分
- 本地匿名 `playerId`
- 开局档案中的昵称和大学名称

轻量设置可以用 `localStorage`：

- 最近一次存档 ID
- 音量、动效开关等轻量偏好
- 是否已看过隐私提示

存档策略：

- 首版不做可见的导出 / 导入备份功能。
- 首版不做云存档。
- 首版不做跨设备同步。
- 浏览器数据被清理后，本地存档和本地 `playerId` 不可恢复。
- 排行榜中已经提交过的总分记录仍保留在云端，但新浏览器无法证明那是原玩家。

技术补充：

- 可以请求浏览器持久化存储权限，降低被浏览器自动清理的概率。
- 不能承诺浏览器本地数据永远不会丢失。
- 不在普通玩家流程中强调备份，避免增加心理负担。

## 开局档案

开局档案必须填写两个输入：

- 昵称
- 大学名称

用途：

- 存档标识
- 游戏称呼和剧情文本
- 排行榜展示

限制：

- 昵称允许重复。
- 大学名称允许重复。
- 昵称建议限制为 `2-12` 个中文字符等价长度。
- 大学名称建议限制为 `2-20` 个中文字符等价长度。
- 不合规时不能开始游戏。
- 前端和后端都做最小字符、长度和敏感词校验。

隐私提示：

`昵称、大学名和总分会用于排行榜展示。`

## 排行榜

排行榜只做总分榜。

展示：

- 显示全服总分 Top 100。
- 显示玩家自己的当前全服排名。
- 排行榜项展示 `昵称 / 大学名称 / 总分`。

总分口径：

- 使用 `endings.md` 中的排行榜累计口径。
- 总分 = 已解锁人生结局分 + 已解锁成长成就分 + 重复达成人生结局折算分。
- 成长成就重复达成不重复计分。

提交规则：

- 每个本地匿名 `playerId` 在云端只保留一个历史最高总分记录。
- 只有本地总分高于云端已记录总分时才更新。
- 排行榜不保存完整存档、完整日志、结局明细或成就明细。
- 排行榜提交失败不阻塞游戏。
- 提交失败可以在本地排队，联网后重试。

排序规则：

- 总分高者优先。
- 同分时，较早达到该总分者优先。

云端最小字段：

- `playerId`
- `nickname`
- `universityName`
- `totalScore`
- `scoreVersion`
- `createdAt`
- `updatedAt`

接口形态：

- `GET /leaderboard/top`：读取总分 Top 100。
- `GET /leaderboard/me?playerId=...`：读取当前玩家总分和全服排名。
- `POST /leaderboard/score`：提交更高总分。

轻量限制：

- 不做登录。
- 不做好友榜。
- 不做赛季榜。
- 不做严格防作弊。
- 不做举报系统。
- 不做后台管理页面。
- 做基础格式校验、长度限制、敏感词限制和提交频率限制。

## 离线与联网

核心游戏必须可以离线游玩。

离线时：

- 可以开始新局。
- 可以读写本地存档。
- 可以解锁本地结局与成就。
- 可以计算本地总分。
- 不展示或不刷新全服排行榜。
- 排行榜提交失败时不影响结局、存档或继续游玩。

联网恢复后：

- 若本地总分高于已提交记录，可以重试提交。
- 排行榜读取只影响展示，不影响游戏规则。

## 版本策略

存档版本：

- 本地存档必须带 `schemaVersion`。
- 小版本字段变化通过迁移函数处理。
- 大版本破坏性变化时，可以提示旧存档不兼容并要求新开局。

排行榜版本：

- 排行榜记录必须带 `scoreVersion`。
- 如果未来总分算法大改，使用新的 `scoreVersion`。
- 不同 `scoreVersion` 的排行榜不混排。

内容 ID：

- 行动、事件、课程、路线、结局、成就等 ID 发布后保持稳定。
- 文案标题可以改，但不能用标题做存档或排行榜主键。

## 部署

首版部署推荐：

- 网站静态资源部署到 `Cloudflare Pages`。
- 排行榜 API 部署到 `Cloudflare Workers`。
- 排行榜数据存储到 `Cloudflare D1`。

发布节奏：

- 先使用平台默认域名测试。
- 正式上线前再绑定自定义域名。
- 首版不接广告和付费。

容量策略：

- 绝大多数游戏流程在浏览器本地完成。
- 云端只处理排行榜读取和总分提交。
- Top 100 可以短时间缓存，减少数据库读取。
- 上万总用户或上万日活的轻量排行榜场景可先用免费额度试运行。

## 测试策略

优先级：

1. 模拟器批量回归。
2. 游戏核心单元测试。
3. 存档迁移测试。
4. 排行榜接口测试。
5. UI 自动化测试。

必须保留：

- 固定 seed 可复现。
- 普通完整通关可复现。
- 四类中途失败可复现。
- 实习、竞赛、路线、结局读取可验证。
- 排行榜总分口径与 `endings.md` 保持一致。

## 逻辑审查

### 本地存档与“不被删除”

结论：没有矛盾，但需要承认限制。

浏览器本地存档可以做到自动保留和下次打开继续玩，但不能保证永远不丢。玩家清理浏览器数据、换设备、使用无痕模式或浏览器策略回收，都可能导致本地数据丢失。当前定案是不展示备份功能，也不做云存档，因此这是接受的产品取舍。

### 无登录与“我的排名”

结论：没有矛盾。

“我的排名”依赖本地匿名 `playerId`。只要浏览器数据还在，就能查询云端对应记录。浏览器数据清除后会生成新 `playerId`，旧排行榜记录保留，但新本地身份无法找回旧记录。

### 昵称大学名与既有开局档案

结论：需要统一命名。

开局档案统一使用“昵称”和“大学名称”。技术实现中分别映射为 `nickname` 与 `universityName`。两者都不提供数值加成，不参与路线和结局门槛，但会用于排行榜展示。

### 轻量化与 UI 动效

结论：没有矛盾。

轻量化指首版不引入重后端、账号、云存档、多榜单和游戏引擎；未来 UI 仍可使用 React 与 GSAP 做舒服的转场、日志、弹窗和状态反馈。动效是表现层，不改变游戏核心边界。

### 网页优先与移动端可玩

结论：没有矛盾。

首版按桌面网页体验设计节奏和布局，代码层采用响应式适配，让手机浏览器可以玩。不追求原生移动端体验，也不单独维护第二套代码。

### 排行榜轻量化与总分累计

结论：没有矛盾。

云端只保存总分摘要，不保存完整图鉴明细。总分明细和完整历史由本地存档负责。排行榜只是展示结果，不作为规则源。

## 第一阶段目标

第一阶段不做 UI 设计，不写 UI 代码。

第一阶段开发前应完成：

- 本文作为技术架构源。
- 明确 `game-core`、存档、排行榜和部署边界。
- 后续代码搭建只围绕这些边界展开。

第一阶段真正写代码时，应先搭建：

- TypeScript 项目结构。
- 独立游戏核心模块。
- IndexedDB 存档适配层。
- 总分排行榜 API 契约。
- 固定 seed 的验证与回归脚本。
