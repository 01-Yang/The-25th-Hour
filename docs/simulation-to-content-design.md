# 模拟结论回写文档

最后更新时间：`2026-05-26 +08:00`

## 这份文档是干什么的

这份文档只做一件事：把这一轮跑局模拟里已经稳定、可以长期保留的内容结论，整理成可并回主线的文档口径。

它不是：

- 不是模拟器使用说明
- 不是临时调参记录
- 不是代码实现文档
- 不是细模块文案草稿堆放处

它要回答的是：这一轮模拟之后，主线文档到底应该稳定成什么样。

## 本轮可并回主线的结论

### 1. 竞赛口径收口

竞赛玩家可见层不再保留过细拆分，统一收口为：

- 竞赛结果弹窗只保留 `未获奖 / 三等奖 / 二等奖 / 一等奖`
- 奖金提示统一为：`竞赛奖金 ￥{amount}`
- 删除“竞赛简历写入提示”
- 删除“竞赛成就提示”

需要保留细分的只有规则层信息，例如具名赛事、投稿门槛、概率分档、奖金倍率；这些继续留在规则与数值文档，不再进入玩家高频可见文案层。

### 2. 实习口径收口

实习玩家可见层不再保留过细阶段提示，统一收口为：

- 成功：`你收到了一封实习录取邮件。`
- 失败：`你盯着那封拒绝信看了很久。`

同时：

- 删除其他“入职”“结束总结”“额外提示”类文案口径
- 保留实习期间三周短事件
- 每条短事件保留对应的轻量属性加成

这意味着实习的玩家体验结构现在很明确：`申请结果 -> 三周过程短事件 -> 路线读取其价值`。

### 3. 实习价值改为累计口径

建筑就业路线读取实习时，不再用“拿过的最高档实习”来理解，而改为累计 `实习价值`：

- 普通实习：`+1`
- 强事务所实习：`+2`
- 名企实习：`+3`

这里的含义是：实习价值反映的是累计履历厚度，而不是一次最高光经历。

### 4. 建筑就业相关文档同步累计颗粒度

与实习相关的主线文档，统一按累计 `实习价值` 写条件，不再混用“任意实习 / 名企实习 / 最高档实习”这种旧口径。

当前主线文档应统一成这类表达：

- `实习价值 >= 1`
- `实习价值 >= 2`
- `实习价值 >= 3`

### 5. 名企不是默认叙事中心

这一轮模拟已经说明：名企实习不是默认结果。

因此主线内容必须承认：

- 普通实习需要正式文案重量
- 强所实习需要正式文案重量
- 拒信也要成为正式反馈的一部分

不能把实习系统写成“只有名企成功才算有内容”。

### 6. 竞赛奖金不是主要成长来源

竞赛奖金可以存在，但它不是竞赛内容的主价值。

竞赛更重要的作用是：

- 给玩家一个“作品走出专教”的节点
- 提供履历确认感
- 提供路线读取时可理解的背景
- 提供身份成长感

所以主线文案不应把竞赛写成刷钱系统。

## 已复用的模拟结论

本轮不是从零重跑全部模拟，而是复用前一轮已经覆盖、且没有被累计 `实习价值` 改动直接影响的基线结论。

复用方式是：

1. 把已经跑通过的结论当作基线证据，不重复消耗时间大面积重跑。
2. 只补跑这次规则变更会影响的路径。
3. 如果补跑发现影响旧流程，再修实现并重跑对应回归。

可以复用的旧结论包括：

- 普通固定 seed 可以完整跑满 `60` 周并进入稳定毕业。
- 四类中途失败可以复现：生活费断裂、被迫停学、压力失控、连续评图失败。
- 竞赛可以输出投稿、未获奖、三等奖、二等奖、一等奖和奖金。
- 实习申请、录取、拒绝、三周完成和最终档位分布已经可统计。
- 后期路线可以完成大四意向、大五正式参与、隐藏判定、毕业揭示。
- 批量报告可以输出竞赛、实习、路线诊断和路线目标矩阵。

这些结论仍然成立；本轮只围绕累计 `实习价值` 做补充验证。

## 本轮新增验证结论

本轮新增实现与验证重点是：累计 `实习价值` 已经从文档口径落实到模拟器实现。

实现口径：

- 实习完成后，`internshipValue` 不再取最高档，而是累计加分。
- 普通实习完成贡献 `+1`。
- 强事务所实习完成贡献 `+2`。
- 名企实习完成贡献 `+3`。
- 建筑就业路线统一读取累计 `internshipValue`。
- 实习申请进阶不读取累计分，而读取最高已完成档位，避免 `普通 + 强所 = 3` 被误判为已经拿过名企。

本轮补跑命令：

```bash
npm run sim:verify:internship-value
npm run sim:verify:internships
npm run sim:verify:route-targets
npm run sim:verify
npm run sim:report -- --count 100 --seed 1 --events true --strategies normal,architecture_job
```

补跑结果：

- `sim:verify:internship-value` 通过：累计值等于所有已完成实习记录之和，多段实习可以超过单次最高档价值。
- `sim:verify:internships` 通过：同 seed 可复现，实习申请/拒绝/完成流程仍有效，普通批量仍可触达名企。
- `sim:verify:route-targets` 通过：建筑就业各目标仍能绑定、跑满、进入最终解析；`master_studio` 已按累计 `实习价值 >= 3` 验证。
- `sim:verify` 通过：普通完整通关和四类中途失败边界仍可复现。
- 100 局补跑通过：`normal` 与 `architecture_job` 均无异常种子。

本轮补跑里曾暴露一个真实问题：累计 `1 + 2 = 3` 后，旧实现会把它当作已达到名企档位，导致系统不再申请名企。现在已拆分为两个概念：

- 路线读取：看累计 `实习价值`。
- 实习进阶：看最高已完成实习档位。

这个拆分是本轮实现对齐里最重要的结构结论。

## 这份文档怎么用

你后面用它，按下面这个顺序就够了：

1. 先把这份文档当成“本轮结论清单”。
2. 检查 `content-plan.md / events.md / numbers.md / endings.md` 是否都已经按这里的口径统一。
3. 如果只沉淀内容口径，可以发 docs-only PR。
4. 如果继续在同一个 PR 里推进实现，就把模拟器代码和验证结果也纳入 PR 范围。
5. 合并后，把 `main` 作为后续内容设计和实现对齐的统一依据。

简单说，这份文档不是给玩家看的，也不是给程序直接读取的；它是给你和后续协作者看的“这轮模拟到底沉淀了什么”。

## 并回 main 的目的是什么

并回 `main` 的目的，不是为了把模拟代码带回去，而是为了把已经确认的内容口径沉淀到主线。

它的价值有三点：

1. 让主线文档停止继续引用那些已经被你删掉的细模块口径。
2. 让后续写正文、做实现、补事件的人都以同一套颗粒度为准。
3. 把“实验分支里的阶段性结论”变成“主线认可的正式结论”。

你可以把它理解成：

- 模拟分支负责试错
- docs-only PR 负责沉淀结论
- `main` 负责保存当前认可版本
