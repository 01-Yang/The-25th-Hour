# 开发前系统收敛冻结口径

最后更新时间：`2026-05-16 +08:00`

本文记录两轮核心架构拷问后的冻结结论。所有条目均已确认采用，用于约束后续开发、调参、内容补全和 UI 落地。

## 使用范围

- 本文只记录系统边界、结算口径、路线读取和开发前冻结原则。
- 具体系统规则仍以 `systems.md` 为准。
- 具体数值、公式、门槛仍以 `numbers.md` 为准。
- 具体事件文本仍以 `events.md` 为准。
- 具体人生结局和成长成就仍以 `endings.md` 为准。
- 若后续讨论与本文冲突，默认按本文处理，除非明确发起新一轮架构变更。

## 总冻结原则

首版不再新增主系统。后续工作只允许进入四类：

- 实现落地：把已定规则写成程序状态机、配置表和 UI 流程。
- 内容补全：补事件、题库、结局文本、课程和角色内容。
- 数值调参：基于跑局模拟调整收益、门槛、权重和失败率。
- 体验打磨：调整展示顺序、提示文案、置灰逻辑和反馈节奏。

## 主循环与阶段

1. 首版主循环冻结为 `5` 年、`60` 周、每周 `3` 次行动、每学期 `1` 个课题、学期末进入评图阶段。
2. 常驻行动池冻结，不再新增主行动；后续只允许通过事件、商店、角色技能和路线节点改数值。
3. 每学期第 `5` 周固定为模型周，第 `6` 周固定为赶图周，时间结构不动态漂移。
4. 每学期课题只保留 `进度` 和 `作品质量` 两个临时值，学期结束后清零。
5. 暑假写生只保留固定节点，不扩成暑假自由行动周。

## 资源与结算

1. 统一结算顺序冻结为：行动、事件、商店、实习结算 -> 金钱即时失败检查 -> 每周自动恢复 -> 高危限制更新 -> 周推进。
2. 所有数值变化必须进入同一个结算器，统一处理上下限、软上限、失败检查和结算日志。
3. 玩家每次行动后都应看到明确数值变化，不只给叙事反馈。
4. 基础资源和长期属性全部透明展示；只隐藏最终路线结果、考试答对数、内部折算分和概率结果。
5. 金钱是硬约束资源：余额不足时禁用花费行为，任意结算后金钱归零立即触发 `生活费断裂`。

## 状态与失败

1. 高危状态只限制行动选择并给提示，不直接触发人生结局。
2. 除 `生活费断裂` 外，所有失败结局触发前都必须有明显预警。
3. `压力失控` 按连续 `2` 周周末仍 `>90` 判定；中途降下来则连续计数清零。
4. 精力为 `0` 不立刻失败；只有 `精力 = 0` 且 `压力 > 90`，并在周末结算时仍未脱离，才触发 `被迫停学`。
5. 评图单次 `F` 不触发退学，只造成 GPA 和路线风险；连续 `2` 次 `F` 才触发 `连续挂科退学`。

## 培养系统边界

1. 作品集只保留一个总值，不拆子分；竞赛结果不直接增加作品集。
2. 导师不做认可度、好感度或独立关系条，只影响阶段任务、汇报偏好、事件倾向和路线倾向。
3. 大学课程保持轻量系统，只影响学期末答题、GPA 修正和事件倾向。
4. 竞赛、实习、建筑生的万里路、AI 相关经历都只作为路线加权、门槛素材或简历记录，不反向改变主循环结构。

## 商店、竞赛、实习

1. 商店只做轻量修正，不做库存、背包、耐久或 build 系统。
2. `星巴克会员周卡` 是唯一允许让下一周行动次数从 `3` 变为 `4` 的例外，且同周购买不叠加。
3. 竞赛固定为单作品唯一投稿、即时公开结果、奖金和简历记录分开结算。
4. 实习从大二上开放，每学期最多申请 `1` 次，录取后固定持续 `4` 周。
5. 实习不占行动槽，但持续期间每周固定结算收益与代价。

## 随机事件体系

1. 事件池只保留 `普通随机事件 / 交互事件 / 模型周事件 / 万里路事件`。
2. 课堂、导师、AI 和地点只做事件语境标签，不新增独立事件池。
3. 事件程序主键必须使用 `event_id`，标题只作为展示文本。
4. 普通事件默认不直接修改 `GPA / 作品集 / 个人简历 / 竞赛结果`；确需修改时必须进入明确白名单。
5. 随机事件抽取首版只使用权重和冷却，不做复杂导演系统或动态剧情树。
6. 交互事件固定只有 `2` 个显式选项，首版不写隐藏第三结果。
7. 模型周事件只影响模型、作品质量、状态或金钱，不直接改路线和长期履历。
8. 建筑生的万里路固定顺序推进地点，不开放自由选点。

## AI 语境

1. AI 只作为时代语境和后期路线条件存在，不做独立 AI 主系统。
2. AI 不做技能树、阵营或第二套能力分。
3. AI 相关经历只由关键 AI 事件、关键选择和少数里程碑产生。
4. 玩家完全不碰 AI 也必须能完整通关。

## 后期路线与结果揭示

1. 大四上第 `1` 周强制锁定后期路线，未锁定时不能继续周行动、实习、竞赛投稿或路线报名。
2. 每局只能锁定 `1` 个后期方向，锁定后不能改线，也不能跨路线补报名。
3. 锁定路线后，UI 应隐藏或置灰其他路线入口，避免玩家误以为还能并行。
4. 路线失败后不跨线补救，只读取该路线回落结局或状态毕业结局。
5. 考试、申请、投递和路线成败结果统一隐藏到大五下毕业设计答辩后揭示。
6. 最终结局读取优先级冻结为：中途失败 -> 没毕业 -> 已锁定路线结果 -> 状态毕业。
7. 强路线永远不是失败条件；没达强路线时，只要完整毕业，仍能进入普通毕业或对应回落闭环。

## 开发前验证

1. 首版必须先做跑局模拟，再决定是否改数值门槛。
2. 强路线触发率、普通通关率和中途失败率都必须通过模拟验证。
3. 随机事件权重、冷却和正负向体感必须通过模拟验证。
4. 画图、赶图、质量类行动的成长曲线必须通过模拟验证。
5. 如果模拟结果不理想，优先调数值、门槛、权重和提示，不凭感觉新增主系统。
