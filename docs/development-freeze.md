# 开发前系统收敛冻结口径

最后更新时间：`2026-05-20 +08:00`

本文记录核心架构拷问与开发落地拷问后的冻结结论。所有条目均已确认采用，用于约束后续开发、调参、内容补全和 UI 落地。

## 使用范围

- 本文只记录系统边界、结算口径、路线读取和开发前冻结原则。
- 具体系统规则仍以 `systems.md` 为准。
- 具体数值、公式、门槛仍以 `numbers.md` 为准。
- 具体事件文本仍以 `events.md` 为准。
- 具体人生结局和成长成就仍以 `endings.md` 为准。
- 若后续讨论与本文冲突，默认按本文处理，除非明确发起新一轮架构变更。

## 总冻结原则

首版不再新增主系统。后续工作只允许进入四类：

- 实现落地：把已定规则写成程序状态机、配置表和 UI 流程。
- 内容补全：补事件、题库、结局文本、课程和角色内容。
- 数值调参：基于跑局模拟调整收益、门槛、权重和失败率。
- 体验打磨：调整展示顺序、提示文案、置灰逻辑和反馈节奏。

核心体验口径冻结为：

1. 首版是“建筑生人生模拟器”，不是纯建筑设计训练模拟器；建筑专业是压力、成长和结局的载体。
2. 每周主要决策压力来自 `3` 次行动机会的分配，不来自复杂菜单、隐藏系统或多路线并行管理。
3. 普通通关应对认真游玩的玩家稳定可达；强路线保持稀有，中途失败要有压迫但不高频。
4. 评图是每学期最核心的结算；进度回答“有没有做完”，作品质量回答“做得好不好”，两者继续拆开。
5. 结局读取保持 `1` 个主结局 + `1` 句状态收尾，不扩成多结局拼贴。

## 主循环与阶段

1. 首版主循环冻结为 `5` 年、`60` 周、每周 `3` 次行动、每学期 `1` 个课题、学期末进入评图阶段。
2. 常驻行动池冻结，不再新增主行动；后续只允许通过事件、商店、角色技能和路线节点改数值。
3. 每学期第 `5` 周固定为模型周，第 `6` 周固定为赶图周，时间结构不动态漂移。
4. 每学期课题只保留 `进度` 和 `作品质量` 两个临时值，学期结束后清零；大五上和大五下作为同一个毕业设计例外，进度与作品质量连续继承，且大五毕业设计的两项上限均为 `200`。
5. 暑假写生只保留固定节点，不扩成暑假自由行动周。

## 资源与结算

1. 统一结算顺序冻结为：行动、事件、商店、实习结算 -> 金钱即时失败检查 -> 每周自动恢复 -> 高危限制更新 -> 周推进。
2. 所有数值变化必须进入同一个结算器，统一处理上下限、学期成长上限、失败检查和结算日志。
3. 玩家每次行动后都应看到明确数值变化，不只给叙事反馈。
4. 基础资源和长期属性全部透明展示；只隐藏最终路线结果、考试答对数、内部折算分和概率结果。
5. 金钱是硬约束资源：余额不足时禁用花费行为，任意结算后金钱小于 `0` 立即触发 `生活费断裂`；金钱等于 `0` 不触发失败。

## 开发落地冻结

1. 游戏状态必须有唯一真相源；长期值、学期临时值、路线锁定、事件记录、隐藏结果和成就解锁都写入统一 `GameState`。
2. 时间推进必须做成显式阶段状态机，至少覆盖：开局、固定流程事件、学期开始、周行动、周结算、评图、学期结算、年度切换、路线锁定、结局结算。
3. 每次行动、事件、商店购买、实习周结算和评图都必须原子化：产出一次结构化 delta、一次结算日志、一次失败检查。
4. 行动入口统一由可用性解析器判断，返回 `可用 / 置灰 / 隐藏` 与原因文案；UI 不自行复制条件。
5. 每局生成固定随机种子；事件抽取、考试抽题、概率录取、投递结果等都走同一 RNG，方便复现和跑局模拟。
6. 事件、行动、路线、结局、成就等内容 ID 发布后保持稳定；展示标题和文案可以调整，程序主键不随文案变化。
7. 路线考试、申请、投递和概率结果进入内部日志与状态缓存，但不提前展示给玩家。
8. 自动存档点固定为：每周开始、每周结束、评图前、评图后、路线锁定后、结局结算前。
9. 最终路线和结局只由统一结局解析器读取，不允许各系统提前写死人生结局。
10. 首版最小可玩闭环先保证一局能从选角色跑到结局，再扩充 UI、事件和美术表现。

## 状态与失败

1. 高危状态只限制行动选择并给提示，不直接触发人生结局。
2. 除 `生活费断裂` 外，所有失败结局触发前都必须有明显预警。
3. `压力失控` 按连续 `2` 周周末仍 `>90` 判定；中途降下来则连续计数清零。
4. 任意结算后精力小于 `0`，立即触发 `被迫停学`；不额外弹出风险确认。
5. 评图单次 `F` 不触发退学，只造成 GPA 和路线风险；连续 `2` 次 `F` 才触发 `连续挂科退学`。

## 培养系统边界

1. 作品集只保留一个总值，不拆子分；竞赛结果不直接增加作品集。
2. 导师不做认可度、好感度或独立关系条，只影响阶段任务、汇报偏好、事件倾向和路线倾向。
3. 大学课程保持轻量系统，只影响学期末答题、GPA 修正和事件倾向。
4. 竞赛、实习、建筑生的万里路、AI 相关经历都只作为路线加权、门槛素材或简历记录，不反向改变主循环结构。

## 商店、竞赛、实习

1. 商店只做轻量修正，不做库存、背包、耐久或 build 系统。
2. `星巴克会员周卡` 是唯一允许让下一周行动次数从 `3` 变为 `4` 的例外，且同周购买不叠加。
3. 商店不设置推荐商店，只展示完整商品清单、分类和可购买状态。
4. 竞赛固定为每学期最多投稿 `1` 次、单作品唯一投稿、即时公开结果、奖金和简历记录分开结算。
5. 实习从大一开始开放；入口未满足最低门槛时仍展示，并提示触发条件；申请不消耗行动次数，采用概率录取；每学期最多申请 `1` 次，录取后一般持续 `3` 周。
6. 实习不占行动槽，但持续期间每周固定结算收益与代价；跨学期时继承剩余周数，已完成实习的单位不能再次申请。

## 随机事件体系

1. 事件池只保留 `普通随机事件 / 交互事件 / 模型周事件 / 万里路事件`，不再设置 `周结算事件`。
2. 课堂、导师、AI 和地点只做事件语境标签，不新增独立事件池。
3. 事件程序主键必须使用 `event_id`，标题只作为展示文本。
4. 普通事件默认不直接修改 `GPA / 作品集 / 个人简历 / 竞赛结果`；确需修改时必须进入明确白名单。
5. 随机事件抽取首版只使用权重和冷却，不做复杂导演系统或动态剧情树。
6. 交互事件固定只有 `2` 个显式选项，首版不写隐藏第三结果。
7. 模型周事件只影响模型、作品质量、状态或金钱，不直接改路线和长期履历。
8. 建筑生的万里路从大二上开放，固定顺序推进地点，不开放自由选点；每次消耗 `2` 次行动机会；可在模型周或赶图周使用；玩家选择前往当前地点后，必定触发对应万里路事件且不占普通周事件上限；完成 `12` 个地点后入口关闭。

## AI 语境

1. AI 只作为时代语境和后期路线条件存在，不做独立 AI 主系统。
2. AI 不做技能树、阵营、态度标签或第二套能力分。
3. AI 相关经历只由关键 AI 事件、关键选择和少数里程碑产生。
4. 玩家完全不碰 AI 也必须能完整通关。

## 后期路线与结果揭示

1. 大四上第 `1` 周强制锁定后期路线，未锁定时不能继续周行动、实习、竞赛投稿或路线报名。
2. 每局只能锁定 `1` 个后期方向，锁定后不能改线，也不能跨路线补报名。
3. 锁定路线后，UI 应隐藏或置灰其他路线入口，避免玩家误以为还能并行。
4. 路线失败后不跨线补救，只读取该路线回落结局或状态毕业结局。
5. 考试、申请、投递和路线成败结果统一隐藏到大五下毕业设计答辩后揭示。
6. 最终结局读取优先级冻结为：中途失败 -> 没毕业 -> 已锁定路线结果 -> 状态毕业。
7. 强路线永远不是失败条件；没达强路线时，只要完整毕业，仍能进入普通毕业或对应回落闭环。

## 开发前验证

1. 首版必须先做跑局模拟，再决定是否改数值门槛。
2. 首轮跑局模拟验收目标：普通通关率 `65%-80%`，中途失败率 `10%-20%`，强路线总触发率 `8%-15%`。
3. 强路线触发率、普通通关率和中途失败率都必须通过模拟验证。
4. 随机事件权重、冷却和正负向体感必须通过模拟验证。
5. 画图、赶图、质量类行动的成长曲线必须通过模拟验证。
6. 最小可玩闭环优先做纯文本或低 UI 版本，先验证 `选角色 -> 60 周 -> 评图 -> 路线锁定 -> 结局` 能跑通，再扩充正式 UI。
7. 首版事件权重配置先以粗权重跑通，不要求一开始精细到每条事件；每条事件先满足 `event_id / pool / phase / result 或 options / base_weight / cooldown_weeks` 即可进入模拟。
8. 概率结果不向玩家展示精确百分比，只展示 `机会较低 / 有一定机会 / 很稳` 这类模糊分档。
9. 高危状态 UI 采用强提醒但不打断流程：状态区呼吸、行动置灰和原因文案优先，不做反复确认弹窗。
10. `没毕业` 结局暂不扩写；先保留最小必要文案，待跑完整局体验后再判断是否补情绪重量。
11. 开发第一阶段完成标准不是内容全部录完，而是固定 seed 下能稳定复现一局，并输出完整结算日志、最终结局和关键统计。
12. 如果模拟结果不理想，优先调数值、门槛、权重和提示，不凭感觉新增主系统。
