# 待验证问题

最后更新时间：`2026-05-25 +08:00`

本文只记录仍需实现验证、跑局模拟或人工审核的问题。已确认口径写回 `PRD.md / systems.md / numbers.md / content-plan.md` 等规则源；历史原因写入 `decision-log.md`。

## 需人工审核

以下问题暂不改动核心数值、标题或正文，等待确认后再写回规则源。

1. `AI 相关事件`保底次数尚未落到 `numbers.md`。
   - 现状：`events.md` 已标记为待审核；`numbers.md` 只有 AI 相关经历来源，没有 AI 事件保底次数。
   - 建议：大一下到大三下每学年最少触发 `1` 次 AI 相关事件，总计 `3` 次；计入对应普通随机事件或交互事件目标，不另开 AI 事件池。

2. `混得入` 被动 `浑水摸鱼` 的触发与抵消口径不够清楚。
   - 现状：效果写作是“每学期 `1` 次低进度或低作品质量状态下，免 `2` 次随机事件或交互事件造成的额外压力惩罚”，同时出现 `1` 次触发和 `2` 次抵消。
   - 建议：每学期最多触发 `1` 次；触发条件为评图前当前进度 `< 70` 或作品质量 `< 45`；触发后抵消接下来最多 `2` 次来自普通随机事件或交互事件的正向压力增量。

3. 固定自动生活费是否可以导致 `生活费断裂` 需要确认体感。
   - 现状：金钱高危只预警和置灰花费行为；任意结算后金钱 `< 0` 会立即读取 `生活费断裂`。
   - 建议：保留固定生活费可导致失败，但在 UI 中至少提前 `1` 周触发金钱高危提示；自动生活费属于可预警的固定结算，不属于不可控文本事件判死。

4. 事件权重的额外修正公式尚未完全量化。
   - 现状：基础事件量、分池目标、普通随机事件连续概率、模型周负面权重修正已量化；但语义重复、同类过密、大四路线焦虑、大五感动事件、`分享会` 大四加权仍只写为“额外权重待审核”。
   - 建议：首版只使用 `base_weight / cooldown_weeks / repeatable / phase / tags`；若要保留额外修正，建议统一为“命中对应阶段或语境时 `base_weight +20`，同类事件连续出现后下一次同类 `base_weight -20`，最低不低于 `10`”。

5. 雅思考试的答题折算为雅思分数尚未量化。
   - 现状：`numbers.md` 写每年 `1` 次、每次 `15` 题，但没有答对题数到 `6.0 / 6.5 / 7.0 / 7.5` 的折算表。
   - 建议：`15` 题中答对 `9 / 10 / 12 / 13` 题分别折算为 `6.0 / 6.5 / 7.0 / 7.5`；低于 `9` 题视为低于 `6.0`。

6. 成就 `天命人` 的 `S 级别角色` 没有角色稀有度来源。
   - 现状：角色池只定义具体角色和初始数值，没有角色稀有度或 S 级标签。
   - 建议：如果不打算做角色稀有度，改为历史游戏开局抽到 `柯布西耶继承者`；如果要保留 S 级概念，则需要在角色池补 `rarity` 字段。

## 实现验证

- 随机事件配置需要落成 `event_id / pool / phase / result 或 options / base_weight / cooldown_weeks` 等字段，并验证单局事件量稳定落在目标区间附近。
- 高危状态 UI 需要验证状态区呼吸、首次高危弹窗、行动置灰和原因提示，尤其是精力高危、压力高危、双高危三种情况。
- 大四上第 `1` 周路线提醒必须是非强制流程；未正式参与方向时，周行动、实习、竞赛投稿、商店和常规推进仍可继续，路线报名、考试、投递和申请不应自动发生。
- 大五路线考试阶段与大五下答辩后的结局结算之间，需要补齐“隐藏结果缓存 + 统一揭示”的实现流程。
- 保研复试、考研初试、雅思和行测考试都需要实现“选完直接下一题、单题不反馈对错”的答题交互。
- 留学申请、实习录取和岗位投递内部判定都需要走固定随机种子；玩家只看到模糊分档或等待结果，不显示精确概率和内部结果。
- `星巴克会员周卡` 是唯一的下周 `+1` 行动次数例外，需要验证同周多次购买不能叠加。
- `基础服务——为人民服务` 判定需要放在 `继续备考` 前，避免被失败兜底吞掉。

## 跑局调参

- 画图、赶图与质量类行动需要跑一轮学期模拟，确认 `软件技术` 和 `设计水平 + 创意审美` 的修正不会让中后期收益失衡。
- 六项属性轻衰减、商店成长商品和实习收益需要验证，避免大二、大三就把关键属性推满。
- 四项竞赛需要模拟验证三等奖、二等奖、一等奖的实际出现率是否符合体感。
- 导师阶段任务需要验证失败惩罚体感。
- `毕业失败——延毕` 结局已保留扩写版本，跑完整局后再判断是否需要调轻或调重。

## 首版闭环

- 首版实现先做纯文本或低 UI 最小可玩闭环，验证 `GameState -> 状态机 -> 结算器 -> 毕业典礼 -> 结局解析器` 能完整从选角色跑到结局。
- 开发第一阶段需要在固定 seed 下稳定复现一局，并输出完整结算日志、最终结局和关键统计。
